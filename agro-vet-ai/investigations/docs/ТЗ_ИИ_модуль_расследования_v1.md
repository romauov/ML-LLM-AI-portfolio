# Техническое задание: ИИ-модуль для расследования ветеринарных инцидентов (v1)

## 1. Введение

### 1.1 Назначение документа
Настоящее техническое задание (ТЗ) определяет требования к разработке ИИ-модуля, предназначенного для автоматизации и структурирования процесса расследования негативных ветеринарных инцидентов (таких как падеж и массовые болезни) в промышленном свиноводстве. Модуль представляет собой программный компонент, использующий LLM (Large Language Model) для анализа ситуации, генерации гипотез и взаимодействия с пользователем-ветеринаром.

### 1.2 Цель проекта
Целью проекта является создание инструмента, который поможет ветеринарам быстрее и точнее определять причины возникновения негативных инцидентов, опираясь на доступные данные и знания, заложенные в ИИ. Модуль должен ускорить процесс расследования, структурировать информацию и формировать опыт в виде структурированных записей для будущего каталога расследованных ситуаций.

### 1.3 Область применения
Модуль предназначен для использования ветеринарными специалистами на свинокомплексах для расследования инцидентов, связанных с:
*   Падежем животных
*   Массовыми заболеваниями

## 2. Функциональные требования

### 2.1 Обработка входных данных
Модуль должен принимать следующие входные данные:
*   **Описание инцидента:** Текстовое описание сути негативного события (например, "6 случаев покусанных хвостов за 3 дня").
*   **Карточка группы животных:** Текстовый документ в формате Markdown, содержащий всю доступную информацию о группе, в которой произошел инцидент. Формат карточки может быть произвольным, но должен соответствовать принятым шаблонам предприятия (см. Пример карточки pig_group.md).

### 2.2 Генерация гипотез
На основе входных данных модуль должен:
*   Проанализировать описание инцидента и карточку группы.
*   Сформировать список потенциальных причин (гипотез) возникновения инцидента.
*   Для каждой гипотезы предоставить аргументацию/обоснование.
*   Ранжировать гипотезы по степени вероятности (оценка производится самой LLM).
*   Предоставить рекомендации по уточняющим действиям или проверке гипотез.

### 2.3 Интерактивное расследование
Модуль должен обеспечивать интерактивный диалог с пользователем (ветеринаром):
*   Представлять гипотезы пользователю.
*   Задавать уточняющие вопросы пользователю, если для анализа не хватает информации (например, "Какова была температура в секции вчера вечером?").
*   Принимать ответы пользователя и корректировать список гипотез на их основе.
*   Возможность пользователя добавлять свои гипотезы, не предложенные ИИ.

### 2.4 Сохранение результата расследования
По завершении расследования (по инициативе пользователя-ветеринара) модуль должен:
*   Запросить у пользователя информацию о результате расследования:
    *   Какая гипотеза была признана верной (если такая есть)?
    *   Если верной гипотезы не было в списке, предоставить возможность пользователю описать её.
    *   Какие действия были предприняты?
*   Сформировать структурированную запись о расследовании в формате, пригодном для сохранения в "Каталог ситуаций" (см. Каталог ситуаций.md).
*   Предоставить возможность вызова внешней функции/инструмента для сохранения этой записи.

## 3. Нефункциональные требования

### 3.1 Интерфейс
*   Основной способ взаимодействия с модулем - REST API.
*   Все входные и выходные данные передаются в текстовом формате (JSON для структурированных данных).
*   Для удобства тестирования и наладки в прототипе будет реализован простой веб-интерфейс, позволяющий вручную вводить данные и просматривать результаты расследования.
*   В финальной версии интеграции в Ветеринарный модуль Агропредикт, веб-интерфейс будет заменен на API-интеграцию с основной системой.

### 3.6 Веб-интерфейс прототипа (для версии v1)
*   **Цель:** Обеспечить удобный способ тестирования и демонстрации функциональности модуля.
*   **Основные страницы/компоненты:**
    *   Форма ввода описания инцидента (текстовое поле).
    *   Форма ввода/загрузки карточки группы животных (текстовая область с поддержкой Markdown).
    *   Панель отображения гипотез с возможностью голосования за наиболее вероятную (для сбора обратной связи).
    *   Интерфейс для ввода ответов на уточняющие вопросы от ИИ.
    *   Форма завершения расследования с указанием верной гипотезы и действий.
    *   Просмотр сгенерированной структурированной записи.
*   **Технологии:** Рекомендуются легковесные фреймворки (например, React, Vue.js или даже простой HTML/JS), обеспечивающие минимальную функциональность.
*   **Аутентификация/Авторизация:** В прототипе v1 не требуется.

### 3.2 Интеграция
*   Модуль должен быть реализован как отдельный сервис с REST API.
*   Внешняя система (например, ERP или специализированное ПО комплекса) интегрируется с модулем посредством вызовов его API.
*   Модуль может вызывать внешние функции для выполнения специфичных действий (например, сохранения файла в каталог).

### 3.3 Технологии
*   Использование LLM (Large Language Model) для обработки естественного языка и генерации ответов.
*   Системные и контекстные промпты должны быть вынесены в конфигурацию или жестко закодированы в логике модуля.
*   Язык программирования и фреймворки определяются разработчиком.

### 3.4 Данные
*   Входные данные: текст (описание инцидента, карточка группы в markdown).
*   Выходные данные: список гипотез (текст), структурированная запись для каталога (например, JSON или YAML).
*   Модуль должен уметь работать с карточками группы произвольной структуры, ориентируясь на ключевые термины и данные.

### 3.5 Системный промпт
Модуль должен использовать системный промпт, который задает контекст и роль ИИ-ассистента. Этот промпт должен включать:
*   Роль: "Вы - эксперт-ветеринар, специализирующийся на промышленном свиноводстве".
*   Цель: "Ваша задача - помочь ветеринару расследовать негативные инциденты, связанные с падежом и болезнями свиней, путем анализа предоставленной информации и генерации обоснованных гипотез."
*   Принципы работы:
    *   Опираться на факты, представленные в описании инцидента и карточке группы.
    *   Задавать уточняющие вопросы, если информация неполна.
    *   Ранжировать гипотезы по степени вероятности, основываясь на доступных данных.
    *   Предоставлять четкие аргументы для каждой гипотезы.
    *   Не выдумывать факты, которых нет во входных данных.
    *   Признавать, когда данных недостаточно для достоверных выводов.
*   Стиль общения: профессиональный, но доступный, ориентированный на практическое решение проблемы.

### 3.5 Управление состоянием
*   Модуль должен поддерживать состояние активного расследования между вызовами API.
*   Сессия расследования идентифицируется уникальным `session_id`, который генерируется при вызове `/investigate` и должен передаваться во все последующие вызовы в рамках этого расследования.
*   Состояние сессии должно включать: исходные данные, историю диалога, текущий список гипотез.
*   Сессии могут иметь ограниченное время жизни (TTL), после которого они автоматически удаляются.

### 3.6 Обработка ошибок
*   Модуль должен корректно обрабатывать и возвращать стандартные HTTP коды ошибок (400, 404, 500 и т.д.) с понятным описанием проблемы в теле ответа.
*   При получении некорректных входных данных (например, отсутствует обязательное поле) модуль должен возвращать ошибку `400 Bad Request`.
*   При обращении к несуществующей сессии должен возвращаться код `404 Not Found`.
*   Внутренние ошибки сервера должны возвращать `500 Internal Server Error`.

## 4. Процесс работы

1.  **Инициация:** Внешняя система передает в модуль описание инцидента и карточку группы.
2.  **Первичный анализ:** Модуль анализирует данные и генерирует первоначальный список гипотез.
3.  **Диалог:** Модуль ведет диалог с пользователем, уточняя детали и корректируя гипотезы.
4.  **Формирование выводов:** Пользователь принимает решение о завершении расследования.
5.  **Завершение:** Модуль запрашивает у пользователя итог расследования и формирует структурированную запись.
6.  **Сохранение:** Внешняя система использует сформированную запись для сохранения в Каталоге ситуаций.

(См. раздел 7 "Сценарии использования" для детального примера.)

## 7. Сценарии использования

### 7.1 Типовой сценарий: Расследование падежа
1.  Ветеринар фиксирует падеж животных и ставит предварительный диагноз (например, "пневмония").
2.  Внешняя система вызывает ИИ-модуль, передавая описание ("Падеж, диагноз: пневмония, 3 головы за 2 дня") и карточку группы.
3.  ИИ-модуль анализирует карточку, генерирует гипотезы (например, "Переохлаждение", "Недостаток витаминов", "Инфекция из соседней секции") с аргументацией.
4.  ИИ задает уточняющие вопросы (о климате, кормлении, состоянии других животных).
5.  Ветеринар отвечает, ИИ корректирует список.
6.  Ветеринар принимает решение. ИИ запрашивает итог.
7.  ИИ формирует структурированную запись для каталога.

## 7. Ограничения и допущения

### 7.1 Первая версия (v1)
*   **Поиск аналогов:** В первой версии функция поиска похожих ситуаций в каталоге НЕ реализуется. Модуль работает только с входными данными конкретного случая.
*   **Накопление опыта:** Модуль формирует структурированную запись для будущего сохранения в каталог, но сам механизм поиска по каталогу отсутствует.
*   **Источники знаний:** Модуль полагается исключительно на знания, заложенные в LLM, и на информацию, предоставляемую пользователем. Он не обращается к внешним базам данных или специализированным ветеринарным справочникам в автоматическом режиме.

### 7.2 Предположения
*   Карточка группы животных содержит достаточный объем информации для проведения базового анализа.
*   Ветеринар готов участвовать в интерактивном диалоге и предоставлять уточняющую информацию.
*   Внешняя система обеспечивает вызов API модуля и сохранение результата в каталог.

## 9. Метрики качества и оценка эффективности (для будущих версий)

Для оценки эффективности работы модуля в будущих версиях могут использоваться следующие метрики:

*   **Точность гипотез:** Процент случаев, когда верная гипотеза находилась в топ-N гипотезах, предложенных ИИ (N=1, 3, 5).
*   **Полнота информации:** Процент случаев, когда ИИ запрашивал действительно необходимую для анализа информацию.
*   **Удовлетворенность пользователя:** Оценка удобства использования и полезности модуля со стороны ветеринаров (опросы, обратная связь).
*   **Скорость расследования:** Среднее время, затрачиваемое на расследование с помощью модуля, по сравнению с традиционным подходом.
*   **Качество структурированных записей:** Соответствие сгенерированных записей заранее определенному формату и полнота заполнения полей.

## 10. Терминология

*   **Инцидент/Ситуация/Проблема:** Негативное событие, требующее расследования (падеж, болезнь, отклонение от нормы).
*   **Описание инцидента:** Краткое текстовое описание сути негативного события.
*   **Карточка группы:** Полный набор информации о группе животных, в которой произошел инцидент.
*   **Гипотеза:** Предположение о причине инцидента, сформулированное ИИ на основе анализа данных.
*   **Расследование:** Процесс анализа инцидента, включающий генерацию гипотез, диалог с пользователем и уточнение информации.
*   **Сессия расследования:** Экземпляр активного расследования, отслеживаемый по `session_id`.
*   **Структурированная запись:** Формализованный отчет о расследовании, предназначенный для сохранения в каталоге ситуаций.

## 11. Дальнейшие шаги и план развития

После реализации первой версии модуля (v1) предполагается следующее развитие:

1.  **Реализация прототипа:**
    *   Создание отдельного веб-приложения с простым пользовательским интерфейсом для тестирования и наладки функциональности модуля.
    *   Веб-интерфейс будет включать формы для ввода описания инцидента и карточки группы, а также отображение гипотез и результатов расследования.

2.  **Интеграция с каталогом ситуаций:**
    *   Реализация функции поиска похожих ситуаций в накопленном каталоге.
    *   Использование исторических данных для повышения точности гипотез.

3.  **Интеграция в Ветеринарный модуль Агропредикт:**
    *   Замена временного веб-интерфейса на полноценную API-интеграцию с основной системой.
    *   Адаптация модуля под архитектуру и стандарты Ветеринарного модуля.
    *   Обеспечение безопасного и стабильного взаимодействия между системами.

4.  **Расширение источников знаний:**
    *   Интеграция с внешними ветеринарными базами данных и справочниками.
    *   Возможность подключения специализированных моделей для анализа конкретных типов данных (например, анализ изображений пораженных органов).

5.  **Улучшение механизмов анализа:**
    *   Внедрение более сложных алгоритмов анализа временных последовательностей событий.
    *   Использование методов машинного обучения для выявления скрытых паттернов.

6.  **Расширение области применения:**
    *   Адаптация модуля для расследования других типов инцидентов (например, отклонения в продуктивности, репродуктивные проблемы).
    *   Поддержка других видов животных, если это потребуется.

7.  **Оптимизация пользовательского опыта:**
    *   Разработка рекомендательной системы для плановых мероприятий по профилактике на основе анализа рисков.
    *   Создание интерфейса для просмотра и анализа каталога ситуаций.
