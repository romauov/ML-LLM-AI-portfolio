## 1. Общее описание

Разработать веб-приложение с интерфейсом ветеринарного ассистента, специализирующегося на расследовании инцидентов в промышленном свиноводстве. Используется упрощенная архитектура с минимальным количеством компонентов и максимальным использованием готовых решений (Langchain, LangServe).

## 2. Цели и задачи

### Цель
Создать веб-интерфейс, который обеспечит ветеринарным врачам удобный способ взаимодействия с ветеринарным ассистентом-расследователем, с сохранением всех функций, присутствующих в консольной версии. (agent-workspace + mcp-server)

### Задачи:
1. Реализовать чат-интерфейс с моделью
2. Обеспечить интеграцию с MCP-инструментами для:
   - Семантического поиска по ветеринарной базе знаний
   - Получения списка источников
   - Получения информации об источниках
   - Извлечения страниц из источников
3. Реализовать систему ведения расследований в форматах, аналогичных консольной версии
4. Обеспечить возможность просмотра и редактирования файлов расследования
5. Реализовать систему управления расследованиями

## 3. Функциональные требования

### 3.1. Интерфейс ассистента
- Основной интерфейс должен включать область чата с моделью
- Поле ввода сообщений
- Возможность отправки сообщений по нажатию Enter
- История сообщений с визуальным разделением на пользовательские и системные

### 3.2. Архитектура Backend

#### 3.2.1. Протокол OpenAI
Backend должен реализовывать совместимый с OpenAI API протокол:
- Endpoint `/v1/chat/completions` для чата
- Поддержка streaming через Server-Sent Events (SSE)
- Поддержка function calling для вызова MCP-инструментов
- Формат сообщений совместимый с OpenAI (role: system/user/assistant/tool)

**Преимущества:**
- Совместимость с существующими фронтенд-оболочками (Open WebUI, LibreChat и др.)
- Стандартизированный протокол
- Возможность легкой замены LLM-провайдера

#### 3.2.2. Дополнительные API endpoints
Кроме стандартного OpenAI API, backend должен предоставлять специфичные для расследований endpoints:
- `GET /v1/investigations` - список всех расследований
- `POST /v1/investigations` - создание нового расследования
- `GET /v1/investigations/{id}` - детали расследования
- `GET /v1/investigations/{id}/files` - список файлов расследования
- `GET /v1/investigations/{id}/files/{filename}` - содержимое файла
- `PUT /v1/investigations/{id}/files/{filename}` - обновление файла
- `PATCH /v1/investigations/{id}/status` - изменение статуса

### 3.3. Модель и LLM
- Подключение к модели Qwen через провайдера OpenRouter
- Использование API-ключа: `{{OPENROUTER_API_KEY}}`
- Базовый URL: `https://openrouter.ai/api/v1`
- Модель: `qwen/qwen3-coder-plus`
- **Вызов MCP-инструментов реализуется на стороне backend**, а не на фронтенде
- Backend преобразует function calls от LLM в вызовы MCP-инструментов

### 3.4. MCP-интеграция

#### 3.4.1. MCP-сервер базы знаний (vetretro)
- Подключение через **SSE транспорт** (предпочтительно) или stdio
- Запускается как отдельный процесс с SSE-сервером
- Backend подключается как HTTP-клиент к SSE endpoint

Конфигурация для SSE:
```bash

python mcp-server/run_sse.py --host 127.0.0.1 --port 8001
```

Переменные окружения:
```

DB_HOST={{DB_HOST}}
DB_PORT=5432
DB_NAME={{DB_NAME}}
DB_USER={{DB_USER}}
DB_PASSWORD={{DB_PASSWORD}}
SIMILARITY_THRESHOLD=0.75
DEFAULT_TOP_K=5
LOG_LEVEL=INFO
```

Предоставляемые инструменты:
- `vet_search` - семантический поиск по базе знаний
- `vet_sources` - получение списка источников
- `source_info` - получение информации об источнике
- `get_pages` - получение контента с конкретных страниц

#### 3.4.2. Backend файловые инструменты (Langchain Tools)
**Упрощенная архитектура:** Файловые операции реализуются как Langchain Tools напрямую в backend, без отдельного MCP-сервера.

**Investigation Manager** - класс для управления файлами расследований:
- Работает только в пределах `agent-workspace/investigations`
- Валидация путей (предотвращение path traversal)
- Создание структуры расследований

**Langchain Tools для файловых операций:**
- `CreateInvestigationTool` - создание новой структуры расследования
  - Параметры: farm_name, animal_type, problem_type, description
  - Создает папку `YYYY-MM-DD_farm-name/`
  - Создает базовые файлы (incident.md, group_card.md, STATUS.md)
- `ListInvestigationsTool` - список всех расследований
- `ListFilesTool` - список файлов расследования
- `ReadFileTool` - чтение файла расследования
  - Параметры: investigation_id, filename
- `WriteFileTool` - создание или полная перезапись файла
  - Параметры: investigation_id, filename, content
- `AppendToFileTool` - добавление контента в конец файла
  - Параметры: investigation_id, filename, content
- `UpdateFileSectionTool` - обновление конкретной секции в markdown
  - Параметры: investigation_id, filename, section_name, new_content
- `TodoWriteTool` - управление задачами (совместимо с Qwen Code)

**Преимущества:**
- ✅ Проще развертывание (2 сервера вместо 3)
- ✅ Нет overhead на HTTP вызовы между процессами
- ✅ Langchain Tools автоматически сериализуются в OpenAI function format
- ✅ Быстрая разработка (~30 задач экономии)

**Важно:** Все файловые операции работают только в пределах директории `agent-workspace` для безопасности.

### 3.5. Система расследований
- Т.е. в других диалоговых системах мы начинаем новый диалог, а в этой системе начинаем новое расследование
- Возможность создания новых расследований с указанием:
  - Название фермы
  - Тип животных
  - Тип проблемы
  - Описание проблемы
- Статусы расследований: активное, завершенное, в ожидании
- Структура файлов расследования, аналогичная консольной версии (подробнее смотри agent-workspace/AGENTS.md).
- Просмотр и редактирование файлов расследования
- Возможность переключения между режимом чата и режимом просмотра файлов расследования, Либо чтобы файлы были видны в левой или правой панели и при клике открывались модальным окном поверх

### 3.6. Механизм автоматического продолжения работы агента (Advanced Feature)

**Цель:** Реализовать механизм, позволяющий агенту самостоятельно определять, завершена ли его работа на текущем шаге, или требуются дополнительные действия перед передачей управления пользователю.

**Описание проблемы:**
В простых чат-системах после каждого ответа LLM управление сразу передается пользователю. Однако в агентных системах (например, Claude Code, Qwen Coder) агент может выполнить несколько инструментов подряд перед тем как запросить информацию у пользователя.

**Примеры:**
- Агент создает файл `hypotheses.md`, затем автоматически читает его для проверки
- Агент выполняет поиск в базе знаний, получает результаты, анализирует их, и только потом задает уточняющий вопрос пользователю
- Агент обновляет несколько файлов последовательно в рамках одного "хода"

**Варианты реализации:**

#### Вариант 1: Инструмент `respond_in_schema` (Рекомендуемый)

**Детальное описание:** См. `docs/auto-continue-mechanism-example.md`

**Оригинальная реализация:** [QwenLM/qwen-code](https://github.com/QwenLM/qwen-code)

⚠️ **Важно:** Перед реализацией необходимо изучить исходный код Qwen Code для понимания деталей агентного цикла и edge cases.

Добавляется специальный инструмент `respond_in_schema`, который позволяет модели явно указать, кто должен говорить следующим - агент (`model`) или пользователь (`user`).

**Как работает:**
1. После каждого ответа агента backend добавляет специальное проверочное сообщение
2. Модель анализирует свой предыдущий ответ по трем правилам:
   - **Rule 1**: Если агент явно заявил о следующем действии или ответ незавершен → продолжить
   - **Rule 2**: Если ответ заканчивается вопросом пользователю → передать пользователю
   - **Rule 3**: Если задача завершена и нет вопросов → передать пользователю
3. Модель вызывает `respond_in_schema` с обоснованием и решением
4. Backend либо продолжает цикл, либо возвращает ответ пользователю

**Преимущества:**
- Не требует изменения основного системного промпта
- Явная структура решения с обоснованием
- Проверочное сообщение не сохраняется в истории

**Пример инструмента:**
```json
{
  "name": "respond_in_schema",
  "description": "Provide the response in provided schema",
  "parameters": {
    "type": "object",
    "properties": {
      "reasoning": {
        "type": "string",
        "description": "Brief explanation justifying the 'next_speaker' choice"
      },
      "next_speaker": {
        "type": "string",
        "enum": ["user", "model"],
        "description": "Who should speak next"
      }
    },
    "required": ["reasoning", "next_speaker"]
  }
}
```

#### Вариант 2: Явный tool `mark_turn_complete`
Добавить специальный MCP-инструмент, который агент вызывает явно когда готов передать управление:
```python
@server.call_tool()
async def mark_turn_complete(reason: str) -> str:
    """
    Агент вызывает этот инструмент, когда завершил все необходимые действия
    и готов передать управление пользователю.
    """
    return f"Turn marked as complete: {reason}"
```

**Недостатки:** Требует, чтобы агент явно помнил о вызове этого инструмента.

#### Вариант 3: Эвристики + respond_in_schema (Гибридный)
Комбинация быстрых эвристик на backend для очевидных случаев:
- Если есть невыполненные tool_calls → продолжить
- Если заканчивается явным "?" → остановиться
- Если есть фразы "далее я...", "теперь прочитаю..." → продолжить
- В остальных случаях → использовать `respond_in_schema`

**Преимущества:** Экономия вызовов LLM в очевидных ситуациях.

**Приоритет реализации:**
Эта функция является **опциональной** и должна реализовываться **после** базового функционала (чат, MCP-интеграция, управление файлами). Сначала нужно убедиться, что базовые возможности работают стабильно.

**Критерии успеха:**
- Агент способен выполнить 3-5 последовательных действий без участия пользователя
- Агент корректно определяет момент, когда нужно запросить информацию у пользователя
- Не происходит "зависаний" (бесконечных циклов вызовов инструментов)

## 7. Системный промпт агента

### 7.1. Базовая основа
Для создания системного промпта ветеринарного ассистента-расследователя используется подход, аналогичный Qwen Code.

**Референсный промпт:** См. `docs/qwen-code-system-prompt.md` - оригинальный системный промпт Qwen Code, который служит образцом для адаптации.

### 7.2. Ключевые адаптации для ветеринарного домена

#### Изменение специализации:
- **Было:** "software engineering tasks"
- **Стало:** "veterinary incident investigation in industrial pig farming"

#### Core Mandates для расследований:
- **Conventions**: Придерживаться структуры файлов и протоколов из `agent-workspace/AGENTS.md`
- **Evidence-Based Reasoning**: НИКОГДА не выдумывать факты. Использовать только:
  - Информацию из описания инцидента (incident.md)
  - Данные карточки группы животных (group_card.md)
  - Результаты поиска в базе знаний (vet_search)
  - Извлеченные страницы из источников (get_pages)
- **Source Citations**: Всегда указывать источник информации с конкретными страницами
- **Explicit Uncertainty**: Явно указывать на недостаток информации вместо предположений
- **Iterative Investigation**: Выдвигать гипотезы → искать доказательства → задавать вопросы → корректировать

#### Primary Workflow: Investigation Process
Адаптация "Plan → Implement → Adapt → Verify":

1. **Analyze**: Изучить описание инцидента и карточку группы
2. **Search**: Использовать MCP-инструменты для поиска в базе знаний
3. **Hypothesize**: Сформировать ранжированные гипотезы с доказательствами
4. **Document**: Обновить файлы расследования (hypotheses.md, knowledge_snippets.md)
5. **Question**: Задать целевые вопросы для уточнения информации
6. **Iterate**: Повторить цикл с новой информацией

#### Task Management
Полностью сохраняется подход с `todo_write`:
- Планирование шагов расследования
- Отслеживание прогресса
- Видимость для пользователя

#### Tools для расследований

**Детальный анализ:** См. `docs/tools-analysis.md` - полный анализ инструментов из Qwen Code и их адаптация для ветеринарного агента.

**Источник инструментов Qwen Code:** `docs/qwen-code-tools.json`

Инструменты разделены на три категории:

**1. Backend-инструменты (реализуются в FastAPI):**
- `read_file` - чтение файлов расследования
- `write_file` - создание новых файлов
- `edit` - редактирование существующих файлов
- `list_directory` - просмотр файлов в папке
- `todo_write` - управление задачами расследования

**2. MCP Knowledge Base (vetretro):**
- `vet_search` - семантический поиск по базе знаний
- `vet_sources` - получение списка источников
- `source_info` - информация об источнике
- `get_pages` - извлечение страниц из источников

**3. MCP Filesystem (mcp-filesystem):**
- `create_investigation` - создание структуры нового расследования
- `list_investigations` - список всех расследований
- `list_files` - список файлов расследования

**4. Специальные инструменты:**
- `respond_in_schema` - для механизма автоматического продолжения работы (опционально)

В виде Langchain PromptTemplate с переменными (investigation_id, workspace_path, etc.):

```python
from langchain.prompts import ChatPromptTemplate

vet_agent_prompt = ChatPromptTemplate.from_messages([
    ("system", """You are a veterinary incident investigator specialized in industrial pig farming.

    Current investigation: {investigation_id}
    Workspace: {workspace_path}

    [... полный промпт с адаптацией из Qwen Code ...]
    """),
    ("placeholder", "{chat_history}"),
    ("human", "{input}"),
    ("placeholder", "{agent_scratchpad}"),
])
```
